# 2 Контрольная Фронтэнд и Бекэнд разработа

В рамках 2КР реализовано 5 практических заданий: 

- 7ПР: Реализованы базовые эндпоинты для входа и регестрации на сайт, а также для работы с товаром:
  `/api/auth/register` POST Регистрация (создание) пользователя
  `/api/auth/login` POST Вход в систему
  `/api/products` POST Создать товар
  `/api/products` GET Получить список товаров
  `/api/products/:id` GET Получить товар по id
  `/api/products/:id` PUT Обновить параметры товара
  `/api/products/:id` DELETE Удалить товар

- 8ПР: Добавлена генерация jwt токена, а именно Access Token, а также authMiddleware для создания защищенных http маршрутов.
  Также добавлен маршрут `api/auth/me` для получения информации о пользователе 
- 9ПР: Добавлена генерация Refresh Token, для формирования нового Access Token, а также добавлен маршрут `api/auth/refresh`, для получения нового рефреш токена, в случае его "устаревания". 
- 10ПР: Проведен полный рефакторинг структуры папок: Использована Onion architecture для бекенда, и FFS для фронтенда, блоки кода разбита на слои, которые между собой взаимодействуют. В рамках самой практики были реализованы страницы для входа и регестрации, React Pages, также было реализовано хранение токенов на фронтенде взаимодействие авторизации между фронтендом и бекендом.
  Также сохранение данных о пользователях и товарах перенесено в СУБД PGSQL, а хранение refresh token перенесено в Redis, где токены хранятся по email пользователя. 
- 11ПР: Сделано RBAC (role-based access control) изменена таблица в pgsql users добавлена колонка role, для сохранения роли пользователя, был изменен axios и добавлен перехватчик для распределения доступа к маршрутам по ролям. 

## Итоги первой контрольной работы Магазин Верхняя одежда
Итогом 2 кр стала реализация полноценной аутентификации и авторизации в магазине верхней одежды, измененная структура бекенда которая поможет проще масштабировать и редактировать функционал, а также измененный формат хранения данных, который обсепечит более удобное хранение данных.
